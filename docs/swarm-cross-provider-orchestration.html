Swarm v2: Cross-Provider
Agent Orchestration

A coordinator agent from one vendor divides the work; worker agents from other vendors build it; findings return; the loop repeats. Claude Code's Task-tool pattern - made heterogeneous. This dossier covers why the current swarm can't do it, proof that every needed capability now exists, and the six phases to build it.

§1Executive summary

Three findings that reframe the feature.

Separate processes are correct, not a compromise

Claude's in-session subagents return every result into the parent's single finite context, with size caps on returns. Separate CLI processes give each agent an independent budget. Vendor diversity and context isolation come from the same decision.

Today's swarm can't do this by construction

Agents launch as one-shot pipes into bare binaries (Get-Content | codex - not codex exec, not claude -p, no permissions, no JSON). No channel into a running agent. Coordinator is only prompted to write tasks.json, then dies. Dead-drop files are not injected by Bridge.

Every capability needed now ships in the CLIs

Resumable multi-turn headless sessions with structured JSON output are documented for claude, codex, and droid. A working open-source reference for the full loop exists (claw-orchestrator).

Why heterogeneous matters: if coordinator and workers were the same vendor, one Claude session could already spawn its own subagents - the feature would be pointless. The value is Claude coordinating while Codex, Grok, Gemini, and Droid build - each role on the model best (or cheapest) for it, each in its own context budget.

phase C non-goal
Do not grow a large TypeScript Autoloop that agent-orchestration-plan.md Phase 3 will delete. Keep the renderer loop minimal; port it as the first engine consumer.
verification methodology
The automated verification phase of the research run failed on a rate limit; every load-bearing claim was then manually re-verified against primary sources. §2How the current swarm works

Codebase findings - the full stack was read: swarmStore, agentSignals, terminalStore, pty.rs, swarm.rs, wizard, SwarmWorkspace, projectStore.

Launch pipeline

  • The 6-step wizard collects a DAG of agents; startSwarmFromWizard creates a second workspace instance of the same folder ("<project> (swarm)") and pins agent panes to it. Same physical directory - no isolation.
  • A React-side dependency scan (swarmStore.ts:823) launches ready agents up to maxParallelAgents. The scheduler lives entirely in the renderer.
  • Each agent's prompt file (mission + role + skills + mailbox/handoff path conventions + markers) is piped into the provider CLI via stdin (pty.rs:424): Get-Content prompt.md -Raw | codex --model …. Stdin is consumed; the shell exits when the CLI finishes.
  • Important: the invocation is the bare binary (codex, claude, gemini, …) - not provider-native headless/exec modes (codex exec, claude -p, droid exec), not structured JSON flags, and no explicit permission posture. Phase A is load-bearing, not polish.
  • Completion = regex-scraping scoped markers on a 512-char rolling tail; already shipped: pty-exit → exitFallbackTransition (clean/unknown → review, non-zero → failed).

Communication is dead-drop files

Bridge does not inject handoff or tasks.json content into dependent prompts. Agents are only instructed (via template text) to open those paths. If the model never opens the file, the edge is empty.

Root causes of "doesn't feel good"

  • A batch pipeline wearing a swarm costume. Plan → exit → build → exit → review. No supervision, no conversation, no mid-course correction.
  • No inbound channel to a live agent. The stdin pipe makes live steering and agent-to-agent talk impossible; the mailbox is write-only theater.
  • Parallel builders share one working tree. Merge-conflict generator; no isolation, no merge-back step.
  • Renderer-owned scheduler with elaborate crash-recovery patchwork because the brain dies with the UI.
  • Keyhole observability: a 4,000-char text tail of one agent, mailboxes polled every 5s.

§3The pattern to replicate

Claude Code's Agent (Task) tool - mechanism verified against official docs.

  • The coordinator calls the Agent tool. Each subagent runs in its own context window; the parent passes only a delegation prompt. VERIFIED
  • Intermediate tool calls stay inside the subagent. Only the final message returns to the parent as the tool result.
  • Parallelism = multiple Agent calls issued in one turn. The coordinator folds results in and decides the next round. That is the loop.
corrected claim - subagent context
An earlier draft repeated a user claim (claude-code issue #10212) that subagents "share the parent's 200K budget." Official docs contradict it: each subagent has its own window. What is true from both sources: every result flows back into the parent's finite context (wide fan-out still bloats the coordinator), and returned results are size-capped (reported 8,192 output tokens; failed large returns still cost 10–15K parent tokens). Design rule: workers write large output to files and return short structured summaries. The conclusion - separate processes give independent budgets - stands, for the corrected reason.

Why not the Claude Agent SDK: it spawns the claude CLI as a subprocess and is locked to Claude models SINGLE-SOURCE - it cannot drive Codex or Grok workers. Cross-vendor orchestration must live outside the SDK, which is exactly the layer Saple Bridge occupies.

Competitive note VERIFIED: Claude Code now documents background agents (parallel sessions, one monitor) and agent teams (sessions that message each other). Anthropic is building the single-vendor version of this feature. The cross-vendor coordinator stays defensible - but the bar is "better than Claude's own teams at the one thing they can't do."

§5How findings return to the coordinator

Three cross-process patterns. Use 1 as the spine, 2 as the live channel.

1

Orchestrator-mediated routing - the spine

Bridge launches a worker headless, parses its JSON result, injects it into the coordinator's next turn via resume. Workers never talk directly; Bridge is the router - deterministic, auditable, human-gateable. This is claw-orchestrator's verified "Autoloop": Planner / Coder / Reviewer, each on an independently selectable engine and model.

2

MCP message bus - the live channel

A bus exposed as an MCP server; any MCP-capable worker joins with zero custom integration (references: MCP Agent Mail, clawo-mcp). Maps directly onto the existing saple-mcp sidecar - which today has zero messaging tools. send_message / fetch_inbox is the missing piece the orchestration plan's Phase 4 already names.

3

File mailboxes - what exists now

Dead-drop files. Only become useful once workers are persistent sessions that can be prompted to check the bus between turns.

The keystone for 1 and 2 is identical: replace the one-shot stdin pipe with resumable sessions.

§7Pitfalls

These will bite. Design for them up front.

  • Context degrades before the limit. Reported quality drop ~50K tokens on 200K windows SINGLE-SOURCE. Coordinator loops need compaction/handoff around 64–80% usage.
  • Result-size caps + coordinator accumulation. Workers write large output to files; return short structured summaries.
  • Human review is the real ceiling. ~5–7 concurrent agents cited optimum; tasks under ~15 min of work have negative ROI after orchestration overhead SINGLE-SOURCE.
  • Rate limits throttle loops. Gemini free tier ~1,000 req/day SINGLE-SOURCE. Per-provider concurrency caps needed.
  • CLI flag drift is constant. --full-auto deprecation, cursor-agent hangs, version-gated behavior. Contain it in a per-provider adapter + version detection + pinned tested versions.
  • Multi-agent pays only on decomposable work SINGLE-SOURCE. The coordinator's first duty is honest decomposition; sometimes the answer is one agent.
  • Headless runs stall on approval prompts. Every worker launch pins an explicit permission posture (--sandbox workspace-write, --auto medium, --always-approve, --permission-mode acceptEdits) or the swarm hangs invisibly.

§8Implementation plan

Six phases; each ships independently and leaves the product working. A→B→C is the value spine; D and E parallel after A; F last. Feeds - doesn't fork - the engine plan in docs/agent-orchestration-plan.md.

A

Provider adapters + structured results

Replace marker-scraping with parsed JSON results; contain flag drift in one module. Required because today launches bare CLIs without exec/print mode, JSON, or permission posture.

  • New src-tauri/src/providers.rs: per-provider adapter - headless_cmd(), resume_cmd(), result_format, parse_result() → AgentResult {text, session_id, cost_usd, is_error, structured}. Replaces the ad-hoc branches at pty.rs:421-441.
  • Verified launch forms: claude -p --output-format stream-json · codex exec --json --sandbox workspace-write --output-last-message <tmp> · droid exec -f prompt.md -o json --auto medium · gemini -p … --output-format json · grok -p … --output-format json --always-approve.
  • PTY keeps streaming to the pane (UX unchanged); a per-attempt extractor parses the final envelope and emits agent-result {paneId, attemptId, result}.
  • swarmStore.ts consumes agent-result as the primary signal; markers + pty-exit stay as fallbacks. Persist providerSessionId on SwarmAgent.
  • Attempt-scoped identity (plan invariant 5): mint a fresh attemptId (and preferably a fresh marker token) per launch/relaunch. Output for attempt N must never complete attempt N+1 - today's seed-time marker reuse is a latent race.
  • Optional cheap win (A or early B): when launching a dependent, Bridge reads upstream outcomes/ and handoffs and injects a short summary into the prompt. Does not replace Phase C; makes the current DAG less hollow.
  • Tests: adapter command-construction + envelope-parsing units with checked-in JSON fixtures; attempt-id isolation (stale attempt output ignored).
EXIT - a template swarm completes with zero marker matches on JSON-emitting providers; markers still complete a custom-provider agent; relaunch cannot be completed by the previous attempt's output.
B

Persistent multi-turn worker sessions

An agent is a session, not a process; anything can be delivered into it mid-run.

  • Rust command agent_send_turn(project, agent_id, message): runs the adapter's resume command as a new attempt; output appends to the same pane; result parsed as in A. Attempt-scoped ids so stale output can never complete a newer attempt.
  • swarmStore.sendTurnToAgent(): rework becomes a turn (feedback lands in the live session with full context) instead of a cold relaunch; operator mailbox messages become delivered turns.
  • Resume failure → fall back to relaunch-with-summary (today's behavior, demoted to fallback). Providers without confirmed resume (grok/gemini/opencode) degrade automatically via resume_cmd → None.
EXIT - reject-with-feedback on a claude/codex/droid worker continues the same session with prior context retained.
C

The coordinator loop

The coordinator stays alive, receives findings, divides work, iterates - the Task-tool loop across vendors.

non-goal (hard)
Do not build a large TypeScript Autoloop that plan Phase 3 will delete. Implement a minimal loop in swarmStore (action parse + route + caps), shaped like the engine's future Command. Port wholesale once Phase 3 lands.
  • Coordinator = persistent claude session with a --json-schema action contract: assign, message, request_worker, review, complete.
  • Bridge drives: worker agent-result → compact summary sent as a coordinator turn (full output stays in files/artifacts) → parse structured_output.actions → execute via the existing scheduler. request_worker keeps its human-approval gate (existing P6 flow); assign/message ride Phase B turns.
  • Guard rails: max coordinator rounds, per-swarm budget cap (--max-budget-usd + summed envelope costs), context watermark ~70% → handoff summary → fresh seeded session.
  • Lives in swarmStore.ts as a thin driver only; ports wholesale into the Rust engine at plan Phase 3.
EXIT - coordinator(claude) → parallel workers(codex+droid) → results return → second round of fixes → complete, with no human input except configured gates. TS loop small enough that plan Phase 3 is a port, not a rewrite.
D

MCP message bus in saple-mcp PARALLEL AFTER A

Agents get a live, provider-agnostic channel; the mailbox becomes an inbox agents actually check.

  • New tools (sidecar currently has zero messaging): send_message {from,to,subject,body}, fetch_inbox {agentId, unreadOnly, waitSeconds} (long-poll - Agent Mail's cost pattern), list_thread. Storage .saple/swarm/messages/.
  • Multi-writer caveat until engine plan Phase 2 lands: append-mostly bus through existing atomic writes; documented gap, fixed for free by Phase 2.
  • Prompts updated: workers check inbox between subtasks; with Phase B, Bridge can push "you have mail" turns.
EXIT - a worker mid-run acts on a message sent after its launch.
E

Worktree isolation + merge-back PARALLEL AFTER A

Parallel builders stop colliding; the swarm runs on an isolated copy and merges back deliberately.

  • Rust: create_swarm_worktree / remove / list via git worktree add, sibling dir ../<repo>-swarm-<id>/ (avoids .saple self-nesting and watcher noise).
  • Wizard toggle "Run in isolated worktree" (default on when >1 editing agent). The swarm's workspace instance points at the worktree - the existing P11 instance mechanism handles the rest.
  • Post-create setup hook runs the configured install command; wizard states the cost honestly (node_modules not shared).
  • Completion: Review room shows worktree diff vs base → merge / PR / discard. Per-agent worktrees after per-swarm proves out; droid workers can use native -w.
  • Pulls forward the item the engine plan deferred - cross-provider parallel builders make its re-entry trigger ("real merge conflicts") a certainty.
EXIT - two builders editing the same file yield two reviewable outcomes, user's own tree untouched.
F

Mission control LAST

Starting a swarm lands you in one screen that shows everything live - inside the existing Swarm room, no new nav entries.

  • On launch: auto-switch to the swarm's workspace instance and the redesigned Swarm room: dependency DAG (existing SwarmGraph) + live terminal grid (reuse TerminalPane; the terminals room's grid and embedded browser panel prove the composition) + right-rail event feed (results, bus messages, coordinator actions, approvals) replacing the 4,000-char tail.
  • Inline approvals wherever the loop needs a human: worker requests, rework, merge-back.
EXIT - a running swarm is fully observable and steerable without leaving the Swarm room.

Cross-phase engineering rules

  • Version pinning + detection: provider store records CLI versions at preflight; adapters declare tested ranges; untested versions warn, never block.
  • Timeouts + leases: every headless attempt gets a deadline; silent-but-alive workers past lease → probe → retry policy.
  • Attempt identity: every launch/relaunch mints a new attemptId; completion events (JSON result, marker, pty-exit) are tagged with it; stale attempt output is ignored (plan Phase 3 invariant 5, enforced early in A/B).
  • Windows-first E2E: every phase exits through a real template swarm on Windows against a scratch repo; macOS smoke on release branches.
  • Engine alignment: A/B/D/E are renderer-safe and engine-agnostic; C is deliberately minimal in TS and becomes the first consumer of the engine's submit/command/observe at plan Phase 3.
  • Do not re-litigate v1 usability: exit fallback, approve/reject, recovery, and in-room tail already exist. New work preserves them as fallbacks while adapters and the coordinator loop take over as the primary path.
