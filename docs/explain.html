Architecture

Under the Hood

The parts worth bragging about

These are real, in-code answers to "tell me about a hard problem you solved."

writeQueue.ts + Rust

Two-layer write safety

Rust writes atomically (temp file + rename under a per-path mutex) so a reader never sees a torn file. A TS enqueueWrite chains writes per file so ordering is preserved and the last save wins. Atomicity and ordering, cleanly separated.

watcher.rs

Live external-edit reconciliation

Agents and the MCP sidecar edit .saple/ behind the app's back. Rust watches the project and emits saple-file-changed; the UI force-reloads the affected store before its next save can clobber the change - reading the live project path so it never reloads into the wrong project.

keychain.rs

Secrets in the OS keychain

API keys never touch JSON, localStorage, markdown, or component state. They live under the OS keychain account saple_bridge_user. A real security boundary, enforced as an architectural rule.

App.tsx

Instant room switching

Heavy, stateful rooms are kept mounted after first visit and toggled with CSS visibility - no xterm dispose/replay, no remount. Light rooms render on demand. Everything is React.lazy + Suspense.

saple-mcp sidecar

Bundled MCP server

A sibling project shipped as a Tauri sidecar. On release it's staged to a stable per-user path so .mcp.json never points at the versioned, ACL-restricted MSIX install directory.

--features ms-store

Store-aware builds

Store builds compile the in-app updater out entirely - the Microsoft Store owns updates for MSIX, and a dormant updater is a policy liability. Non-store builds keep the Tauri updater. One codebase, two channels.

How Everything Connects

One save, traced end to end

Follow a single Kanban edit from a click to the disk and back - it touches every layer.

1

User drags a card

React updates kanbanStore optimistically so the UI feels instant.

2

The store enqueues a write

enqueueWrite('tasks', ...) serializes it behind any in-flight save for the same file.

3

invoke crosses into Rust

write_project_file writes to a temp file and renames it, under a per-path mutex - atomic, never torn.

4

An agent edits the same file

Meanwhile the MCP sidecar writes tasks.json too. watcher.rs notices and emits saple-file-changed.

5

The UI reconciles

React force-reloads kanbanStore from disk so its in-memory copy matches - before the next save. Human and agent stay in sync.

Quick Answers

The questions you'll get asked

Crisp, defensible answers - each backed by the real code.

Why Tauri and not Electron?

Smaller binary and memory footprint - it uses the OS webview instead of bundling Chromium - and Rust for the OS-facing work. It also packages cleanly to MSIX for the Microsoft Store.

Where does data live? Is there a database?

No server or DB. Everything is plain files inside the project's .saple/ folder - JSON for tasks/swarm/sessions, markdown + frontmatter for memory. Secrets are the one exception: they live in the OS keychain.

How do you prevent two writers from corrupting a file?

Two layers. Rust writes atomically (temp file + rename under a per-path mutex) so reads are never torn. A TypeScript write-queue serializes writes per file so ordering is preserved and the last save wins.

How does the UI stay in sync when an agent edits files directly?

A Rust file watcher emits a saple-file-changed event; the frontend force-reloads the affected store before its next save, using the live project path so it can't reload into the wrong project.

How do your different apps talk to each other?

Through a localhost-only, token-authed control endpoint with three operations (capabilities / command / observe), a monotonic event log, resumable observation, and idempotent commands. It's opt-in and default-off.

What was the hardest part?

Getting write-safety and external-edit reconciliation right. The app and autonomous agents write the same files concurrently, so I separated atomicity (Rust) from ordering (TS) and reconcile external edits without clobbering in-flight saves.

What are the limits, and what's next?

Local-first means no built-in multi-device sync - that's the deliberate tradeoff for privacy and simplicity. Cross-machine sync is the natural next feature, and it'd have to preserve the "your data is just files" guarantee.
