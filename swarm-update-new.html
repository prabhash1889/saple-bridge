Swarm Complete Redesign

A from-scratch replacement of the current swarm with real orchestration: event-driven, contract-driven, and subscription-aware. The coordinator coordinates, builders build live, and "completed" means the acceptance command actually passed.

Design goals

Coordinator actually coordinates

Its plan is machine-read and materializes workers. It stays alive, receives results, reacts, and produces the final answer.

Seamless and realtime

File-watcher events replace every 5-second poll. A task completion propagates to the scheduler, the UI, and the coordinator in milliseconds.

Contracts, not conventions

plan.json, verdicts/*.json, outcomes/*.json are parsed, sanitized, and enforced by Bridge. Agent output is untrusted: sanitize, fall back, or escalate - never trust, never hang.

Verified completion

completed means the acceptance command passed, not "markers printed".

Subscriptions are the fuel

Sign-in detection for every provider, auto-assignment of tasks across signed-in CLIs, per-provider concurrency caps and rate-limit cooldowns.

What is kept vs deleted

The current swarm is a hardened substrate wearing orchestration theater. Keep the substrate, delete the theater.

Keep - proven substrate

  • PTY spawn, allowlist, validated inputs, headless exit codes pty.rs
  • Scoped lifecycle markers + rolling-tail detection + exit fallback agentSignals.ts
  • Crash/restart reconciliation + cross-project recovery loadSwarmState
  • Serialized scheduler core, parallel cap, per-agent commit discipline
  • Bounded rework (reworkAgent, attempt budgets)
  • Mailboxes, handoffs, workspace pinning
  • Control-plane records controlPlane.ts
  • Provider readiness store providerStore (extended in Phase 6)
  • Rust file watcher watcher.rs (promoted to event backbone)

Delete - orchestration theater

  • 6-step wizard as the launch flow SwarmWizard.tsx
  • Static roster templates DEFAULT_TEMPLATES, SwarmTemplateEditor
  • Prompt-only tasks.json convention (nothing parses it)
  • 5-second polling loops in SwarmWorkspace
  • Human-approval-only worker requests (requests.json)
  • "All agents done = completed" terminal condition

Contracts

Sanitizer rules: unknown fields dropped; missing id/mission drops the task; unknown deps filtered; cycles rejected; task ids are append-only dedup keys. A malformed plan can never crash the scheduler or launch a bad worker.

// plan.json (coordinator-owned, Bridge-sanitized)
{
  "version": 2,
  "acceptance": { "command": "npm test", "description": "suite green" },
  "tasks": [
    {
      "id": "fe_auth",
      "mission": "Implement login form + token storage in keychain",
      "role": "builder",
      "dependsOn": ["design"],
      "provider": "auto",
      "model": "default",
      "review": true,
      "strategy": "single"
    }
  ]
}

// verdicts/<taskId>.json (reviewer-owned)
{ "taskId": "fe_auth", "verdict": "reject", "feedback": "token stored in localStorage, move to keychain" }

// new lifecycle markers (same scoped-token scheme as [AGENT_DONE:<token>])
[PLAN_READY:<token>]    // coordinator finished initial planning
[PLAN_UPDATED:<token>]  // coordinator appended tasks mid-run (repair wave / dynamic growth)
PHASE 0

Contracts and parsers

pure lib - zero behavior change

The foundation everything else compiles against. No UI or runtime changes ship here.

  1. src/types/swarmPlan.ts - SwarmPlan, PlanTask, Verdict, SwarmWave, AutonomyMode (manual | gated | auto), SwarmEvent union (plan_ready, plan_updated, task_completed, task_failed, verdict_recorded, wave_completed, acceptance_passed, acceptance_failed, escalated).
  2. src/lib/swarmPlan.ts:
    • parsePlan(raw) - full sanitizer per contract rules (mirror parseWorkerRequests / parseAgentOutcome style: drop, never throw)
    • parseVerdict(raw) - only approve/reject accepted; anything else null (parks task for a human)
    • diffPlan(appliedIds, incoming) - returns only new, valid, acyclic tasks
    • Cycle detection: reuse the logic behind validate_dependency_graph, ported to run sync in TS
  3. agentSignals.ts - add scoped PLAN_READY / PLAN_UPDATED regexes to the cached per-token battery; extend mightContainAgentMarker with PLAN_.
  4. Vitest: every malformed-input path (missing fields, wrong types, cycles, duplicate ids, hostile strings), plus marker tests mirroring agentSignals.test.ts.
Done when: typecheck and tests pass; nothing user-visible changes.
PHASE 1

Realtime event backbone

kill all polling

Replace every setInterval with push events from the Rust file watcher.

  1. Rust (watcher.rs): watch_swarm_dir / unwatch_swarm_dir commands - recursive, debounced (~150 ms) notify watcher on .saple/swarm/ emitting a swarm-file-changed {projectPath, relPath} Tauri event. Register in lib.rs.
  2. src/lib/swarmEvents.ts - tiny TS event bus: subscribes once, classifies relPath (plan / verdicts / outcomes / mailbox / handoffs), fans out typed callbacks. Single module-level listener, mirrors startPtyOutputListener lifecycle.
  3. swarmStore: start/stop watching in loadSwarmState / stopSwarm; route classified events to handlers (stubs now; Phases 2-5 fill them in).
  4. SwarmWorkspace.tsx: delete the 5 s worker-request poll and mailbox/handoff polling effects; re-read a file only when its event fires.
  5. Guard: events for a project that is not loadedProjectPath are dropped (P13 recovery already covers away-project catch-up on load).
Done when: editing a mailbox file externally updates the open Swarm room in under a second with zero intervals registered.
PHASE 2

Coordinator-driven orchestration core

the pivot

The swarm starts with a coordinator only; its plan materializes the workers. The wizard DAG dies.

  1. Store rewrite keeping the proven pieces. State adds: plan, appliedPlanTaskIds, autonomy, wave, maxWaves, maxParallel (persisted in state.json). Keep: marker minting, scheduler serialization guard, per-agent commit discipline, reconciliation, mailbox/handoff/outcome plumbing, reworkAgent.
  2. startSwarm(mission, options) replaces startSwarmFromWizard: writes dirs, pins a swarm workspace instance (existing P11 flow), seeds ONE coordinator agent, launches it.
  3. Coordinator prompt (src/lib/swarmPrompts.ts): mission, the plan.json contract verbatim, the acceptance-command requirement, scoped markers incl. [PLAN_READY] / [PLAN_UPDATED], mailbox path.
  4. Plan intake on marker (fast path) or plan.json watcher event (fallback): read, parsePlan, diffPlan, materialize each new task as a SwarmAgent (fresh id + marker, prompt from task.mission + role sections), append applied ids, persist, run scheduler.
  5. Scheduler: existing runAgentScan logic (deps-done, launch under cap) on materialized tasks; plan dependsOn maps through a taskId-to-agentId map in state.
  6. Worker prompts: mission comes from the plan task. Remove the tasks.json paragraph everywhere - the plan IS the assignment.
  7. Failure containment: coordinator exits without a valid plan -> swarm parks in review with the terminal tail visible and one-click "retry planning".
  8. Tests: plan intake (valid / partial / garbage / duplicate re-emit), materialization idempotency, scheduler cascade.
Done when (E2E): mission -> coordinator plans -> workers spawn from plan.json -> complete -> dependents cascade. Kill the app mid-run -> reconciliation recovers.
PHASE 3

Live coordinator

seamless, not fire-and-forget

The coordinator stays alive for the whole swarm and receives results in its own PTY.

  1. pty.rs: new spawn mode interactive_with_prompt - launch the CLI interactively (no stdin pipe, no headless exit wiring), deliver the prompt by writing it to the PTY via the existing writer path once ready. Coordinator panes use this; workers stay headless.
  2. Digest injection: notifyCoordinator(digest) writes a compact results digest (task id, status, outcome summary, verdict) into the coordinator's PTY as a user turn. Triggered on wave_completed and terminal task failures.
  3. Busy/idle tracking: only inject at the input prompt (quiet-output heuristic); queue digests while busy, flush on idle. Observable store state: planning | idle | digesting.
  4. Fallback (per-provider flag in providerMeta): TUIs that reject injected turns get digest-relaunch - fresh coordinator whose prompt embeds all outcomes/verdicts so far. Injection is the optimization; relaunch is the guarantee.
  5. Coordinator crash mid-swarm: exit fallback detects it; auto-relaunch once with the digest prompt; a second crash escalates to human.
Done when: a worker finishes and its summary visibly appears inside the coordinator's terminal within seconds, and the coordinator's next output reacts with prior context intact.
PHASE 4

Automated review gate

verdicts drive rework
  1. Every plan task with review: true auto-generates a reviewer agent depending on it (shared reviewer per wave by default; per-task for large waves). Reviewer prompt: target mission + outcome file + diff-scope instruction + the verdict contract + scoped markers.
  2. On reviewer completion (or verdicts watcher event): parseVerdict.
    • approve -> task approved; dependents treat it as done
    • reject -> auto reworkAgent(taskAgent, feedback) (existing bounded loop; budget exhausted -> existing human escalation UI)
    • missing / garbage -> park the task in review for a human. Never guess.
  3. Autonomy dial per swarm: manual (every transition needs a click - debug), gated (auto-rework, human approves the plan - default), auto (hands-free within budgets).
  4. Dependents of a rejected task stay blocked; existing blocked-cascade logic works once "done" is redefined as done + approved when review:true.
  5. Tests: verdict parse paths, auto-rework trigger, budget exhaustion, garbage-verdict parking.
Done when: a reviewer rejection relaunches the builder with feedback embedded, zero human clicks, at most maxAttempts times.
PHASE 5

Acceptance and repair waves

completed = verified
  1. Acceptance runner: Bridge (never the coordinator - never trust an agent's claim) runs plan.acceptance.command in a headless PTY in the project dir when all tasks are done + approved. Exit 0 = pass. The command is operator-visible before launch.
  2. Pass -> synthesis digest into the coordinator -> final report written -> stored as a control-plane artifact -> swarm completed, report surfaced in the UI.
  3. Fail -> failure-output digest -> coordinator appends repair tasks + [PLAN_UPDATED] -> Phase 2 intake + Phase 4 gate run the repair wave; wave increments.
  4. Guard rails: maxWaves (default 3); a wave with zero new tasks or two consecutive identical acceptance failures (trimmed output hash) short-circuits to escalation; escalation = structured report artifact (waves attempted, remaining failures, diagnosis, proposed next wave) with human choices: one more wave / redirect / stop.
  5. Tests: pass/fail transitions, wave counting, identical-failure short-circuit, escalation payload shape.
Done when: an intentionally broken build triggers a repair wave that fixes it; an unfixable one escalates with a readable report instead of looping.
PHASE 6

Subscription-aware provider layer

spread, cap, cool down
  1. Extend check_provider_signin beyond codex/claude: gemini (~/.gemini/oauth_creds.json), copilot (gh auth status), opencode (opencode auth list); cursor/droid/grok/pi where a probe exists, honest None where not.
  2. Fix the silent prompt hole: providers failing provider_accepts_prompt_pipe (cursor, copilot) must not launch mission-less TUIs as workers - deliver the prompt via the Phase 3 stdin-write path, or refuse the assignment at materialization with a clear per-task status.
  3. Provider assigner (src/lib/providerAssign.ts): resolves provider: "auto" across ready providers (installed + signed-in/keyed + enabled), round-robin weighted by caps; deterministic and unit-tested. Explicit provider wins if ready, else falls back with a status note.
  4. Per-provider concurrency caps (settings, default ~2 per subscription CLI) under the global maxParallel: scheduler counts running agents per provider and holds excess in waiting.
  5. Rate-limit awareness: per-provider rate-limit phrase regexes in providerMeta matched on the signal tail; on hit, mark the provider cooling for N minutes, requeue to another ready provider or hold, surface a badge in the UI.
  6. Tests: assigner distribution, cap enforcement, rate-limit requeue.
Done when: a plan with provider "auto" spreads tasks across every signed-in CLI, respects caps, and a rate-limited provider visibly cools down while work continues elsewhere.
PHASE 7

Mission-first UI

replace the wizard
  1. Launch composer (replaces SwarmWizard and all steps): one screen - mission textarea, autonomy dial, maxParallel / maxWaves, provider preference chips auto-populated from readiness (unready greyed with the reason). Command Palette's pendingWizardMission seeds it.
  2. Plan approval screen (manual/gated): the parsed plan as editable cards - edit mission text, change provider/model, toggle review, delete tasks, see the acceptance command. Approve -> materialize + launch. auto skips straight through.
  3. SwarmWorkspace rebuild: wave timeline header (plan -> build -> review -> acceptance per wave, live states); agent cards with existing status colors + verdict badges, outcome summaries, elapsed time; coordinator strip (planning / idle / digesting + latest digest); escalation panel with the three actions; SwarmGraph kept, reading the same materialized roster, restyled for waves.
  4. Delete: SwarmWizard.tsx, wizard/steps/*, SwarmTemplateEditor.tsx, DEFAULT_TEMPLATES, template persistence in partialize. Role prompt presets survive as a small rolePresets.ts used by the prompt builder.
  5. No new nav: the swarm entry point stays exactly where it is today.
Done when: mission -> glance at proposed team -> launch in under a minute; the room shows a live, self-updating picture with zero manual refresh.
PHASE 8

Patterns, hardening, telemetry

debate + cleanup
  1. Debate strategy: "strategy": "debate", "attempts": N on a plan task -> N parallel builders (isolated dirs or file-scope partitioning) + an auto-generated judge task writing a verdict that picks/merges - the Phase 4 verdict contract reused verbatim.
  2. Telemetry in state.json: waves used, rework counts per task, per-provider launch/failure/rate-limit counts, wall-clock per phase. Feeds escalation reports and tuning; nothing transmitted externally.
  3. Prompt-injection hardening pass: every agent-written file re-audited against its sanitizer; fuzz tests with hostile strings; markers remain the only status channel and stay scoped per agent.
  4. Docs: CLAUDE.md storage table (plan.json, verdicts/), src/AGENTS.md, src-tauri AGENTS.md (watcher + interactive-prompt spawn mode).
  5. Cleanup sweep: delete dead old-swarm code (parseWorkerRequests, resolveWorkerRequest, requests polling, unused types) with reviewable per-phase diffs.
Done when: a debate task produces a judged merge; telemetry shows up in escalation reports; no orphaned old-swarm code remains.

Rollout, risk, and testing

  • Phase order is dependency order. 0-1 are invisible foundations; 2 is the breaking pivot. Between 2 and 7 the composer is a minimal mission box so the app never ships wizard-less AND composer-less.
  • Trust posture everywhere: agent output is untrusted input. Sanitize-or-drop on every read; park-for-human on ambiguity; markers scoped per agent; acceptance is executed by Bridge, never self-reported.
  • Containment: all new files stay inside .saple/swarm/* via existing contained write commands. New Rust surface is small: swarm-dir watcher + interactive-prompt spawn mode, both validated like existing inputs.
  • Crash safety: every new state field lives in state.json and round-trips through existing reconciliation; a mid-wave restart resumes the wave, never restarts it.
  • Test strategy per phase: vitest for every parser/scheduler/assigner path, cargo test for watcher and spawn-mode validators, one real E2E swarm run on Windows before each phase merges.
