Saple Bridge · Swarm v2

From launcher to real orchestration

Today you hand-build the roster and hope the agents cooperate. After the redesign, you state a mission - the coordinator plans, staffs, reviews, verifies, and repairs until the result actually works. All on your existing CLI subscriptions.

mission→ plan→ build→ review→ integrate→ verify→ repair*→ done
The change

Today vs after

The execution substrate (PTY terminals, lifecycle markers, crash recovery) is solid and stays. The orchestration brain is what gets built.

Today Static launcher

  • You design the team: count, roles, models, dependencies - in a 6-step wizard, before any AI reads the code.
  • The coordinator's "plan" (tasks.json) is prompt convention. Nothing parses it.
  • Reviewer runs after builders but cannot reject their work - only you can, by clicking Rework.
  • New workers mid-run need your manual approval, every time.
  • "Completed" = every agent printed its done-marker. Nobody checks it works.
  • No final combined answer.

After Plan-driven orchestrator

  • You give a mission + limits. The coordinator explores the repo and writes the plan.
  • plan.json drives the scheduler. Tasks, roles, and dependencies are machine-read and executed.
  • Reviewer verdicts automatically bounce bad work back to the builder with feedback (bounded).
  • The coordinator grows the plan mid-run; within your budget it's hands-free.
  • "Completed" = the acceptance check passed (tests green, build compiles) - verified, not narrated.
  • A synthesis turn produces the final report; failures trigger repair waves.

The glue

Three small contracts

Agents are subscription CLIs - Bridge can't force their output, so every contract is sanitized on read. Malformed data is dropped or escalated to you. Never trusted, never hung.

plan.json

written by coordinator · parsed by Bridge
{
  "acceptance": { "command": "npm test" },
  "tasks": [{
    "id": "fe_auth",
    "mission": "Login form + token storage",
    "role": "builder",
    "dependsOn": ["design"],
    "review": true
  }]
}

The living DAG. Appending tasks mid-run is one primitive covering both dynamic growth and repair waves.

verdicts/<task>.json

written by reviewer · acted on by Bridge
{
  "taskId": "fe_auth",
  "verdict": "reject",
  "feedback": "token in localStorage,
               move to keychain"
}

Reject → Bridge relaunches that builder with the feedback, automatically, within the attempt budget you set.

outcomes/<agent>.json

written by each worker · already exists today
{
  "summary": "login form done",
  "changedFiles": ["src/Login.tsx"],
  "tests": { "command": "npm test",
             "passed": true }
}

Feeds the reviewer, the coordinator's synthesis turn, and the run records you already see in Review.

Your side of it

You provide three things

The 6-step roster wizard becomes the advanced path. The default path:

1

Mission

What you want done, in your words. The coordinator reads the actual repo before deciding anything.

2

Limits

Max parallel agents, repair-wave budget, and the autonomy dial: approve everything, approve the plan only, or fully auto.

3

Provider preferences (optional)

"Coordinator on Claude, builders on Codex" - or nothing, and Bridge uses whatever CLIs are signed in.

Everything you configure by hand today gets derived

Agent count - from how many parallelizable tasks the plan really has
Roles + dependencies - from the plan's task graph
Provider per agent - from your signed-in subscriptions, spread to dodge rate limits
Model per agent - sensible default per role, one global override
System prompts - generated from each task assignment itself
Mid-run staffing - coordinator adds workers within your limits
Build order

Seven phases, each shippable

Manual mode keeps working the whole way through - it is the permanent fallback and the advanced path. Full detail in swarm-redesign.md.

Phase 0

Contracts & parsers

Plan/verdict types, sanitizers, plan markers, full test coverage. Pure library code, zero behavior change.

every malformed-input path has a test
Phase 1

Plan-driven scheduler

THE PIVOT

Swarm starts with the coordinator alone. Its plan.json materializes workers through the existing scheduler. The plan becomes the assignment; tasks.json convention dies.

mission in → coordinator plans → workers spawn from the plan → cascade completes
Phase 2

Automated review gate

Review tasks auto-generated per plan task. Verdict files drive approve/reject; rejection auto-reworks the builder with feedback. Autonomy dial added.

a rejection relaunches the builder with no human click
Phase 3

Coordinator wake & synthesis

Session resume (claude --resume / codex resume, digest fallback for all providers). Coordinator reacts to results with full memory; synthesis turn writes the final report.

coordinator demonstrably reacts to worker results with prior context
Phase 4

Acceptance & repair waves

The acceptance command must actually pass. Failure → targeted fix-tasks → new wave. Wave budget, no-progress short-circuit, structured escalation report.

a broken build triggers a wave that fixes it; an unfixable one escalates readably
Phase 5

Mission-first UI

Mission + limits + prefs → plan approval screen → launch. Provider auto-assignment from subscriptions. Wave timeline and verdict badges in the workspace.

mission → glance at proposed team → launch, under a minute
Phase 6

Patterns & hardening

Debate strategy (N attempts + auto-judge), per-provider concurrency caps for subscription rate limits, wave/rework telemetry.

a debate task produces N candidates and a judged winner
Why it won't run away

Guard rails

Autonomy is budgeted, never open-ended.

Bounded everything

Rework attempts per task, repair waves per swarm (default 3), parallel-agent caps globally and per provider. Budgets exhausted → a human decides, with a full report.

Untrusted agent output

Every file an agent writes is sanitized on read - bad plans, garbage verdicts, and missing outcomes degrade to "park it for a human", never to a hang or a bad launch.

No stuck loops

A repair wave that adds zero new tasks, or the same failure twice in a row, short-circuits straight to escalation instead of burning subscription quota.

Verified done

The coordinator cannot declare success by narrating it. The acceptance check is a real command that must pass - the same E2E-first standard you hold for bug fixes.

Subscription-native

Agents stay CLIs in PTYs, so Claude Max / ChatGPT plans power the whole swarm at flat cost. Coordination is turn-based (seconds), the price of not burning API credits.

Contained writes

All new state lives under .saple/swarm/* through the existing path-contained commands. No new filesystem surface, no secrets outside the OS keychain.
