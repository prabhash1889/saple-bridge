EXECUTIVE FINDING

Same destination. Opposite starting points.

The products overlap in multi-agent orchestration, but their center of gravity is different.

CNVS makes agents feel tangible. SAPLE makes their work accountable.

The strongest move is not to clone CNVS’s infinite canvas. SAPLE should connect the lifecycle infrastructure it already owns, then borrow CNVS’s best interaction ideas: universal directing, voice as an input layer, visual feedback and safe remote access.

CNVS wins onImmediacy, spatial awareness, visual loops
SAPLE wins onGovernance, review, durable state, provider depth
Best opportunityInteraction layer over SAPLE’s control plane

WHERE EACH WINS

Protect the moat. Borrow the magic.

SAPLE should not trade its verifiable workflow model for a looser clone of CNVS.

SAPLE’s durable advantages

  • Explicit task graphs, worker caps and crash reconciliation.
  • Review gates tied directly to Git operations.
  • Inspectable local state and a broad MCP tool surface.
  • Provider breadth and cross-platform desktop reach.
  • Permissions, incidents, handoffs, artifacts and run history.

CNVS’s interaction advantages

  • Spatial overview makes parallel activity legible at a glance.
  • Voice lowers the cost of directing agents while staying in flow.
  • Embedded browser and screenshots close the visual QA loop.
  • Remote persistence makes the workspace feel always available.
  • Freeform spawning and prompting feels immediate and playful.
INTERNAL GAP

The highest-leverage fix is already inside the repo.

Bridge’s live sessions and saple-mcp’s durable runs are adjacent systems, not one continuous lifecycle.

Bridge launch

Starts a provider terminal and keeps session state in the desktop application.

→
Live work

Status, output and review flow continue locally, but artifacts are generally left empty.

⇢
SAPLE MCP

Separately owns agents, runs, events, context briefs, artifacts and final outcomes.

Observed consequence: review code can look for a test_result artifact, and the type supports artifacts, but Bridge does not currently populate that lifecycle. Launch prompts also do not request MCP project, task or agent briefs.
RECOMMENDED MOVES

Add leverage in this order.

The sequence begins with integration, then makes orchestration feel faster, then expands the feedback and remote surface.

Connect Bridge to the existing MCP lifecycle

Register agents, create/update/finish runs, append events and attach review outcomes.

Foundation / highest leverage

Inject on-demand context briefs

Call project, task and agent brief tools at launch and before bounded retries.

Low UI cost

Build a universal agent composer

Extend the command palette to target one agent, all agents, a reviewer, a task or a new terminal.

Interaction layer

Capture structured outcomes

Turn files, tests, summaries and screenshots into durable artifacts instead of terminal-only output.

Trust layer

Add a bounded review-rework loop

Reviewer feedback can reopen work with an attempt cap and visible history.

Safe automation

Add localhost preview and screenshot-to-context

Keep it loopback-only first; make visual QA an artifact agents can inspect.

CNVS-inspired

Allow approved dynamic worker requests

Agents request workers through MCP; Bridge authorizes and launches them under the swarm cap.

Guarded delegation

Ship SSH terminal presets

Offer saved remote shells without pretending they are full remote workspaces.

Incremental remote

Recommended product position

“A local-first control plane for coding agents—structured enough to trust, immediate enough to stay in flow.”

Borrow CNVS’s interaction quality without erasing SAPLE’s stronger differentiator: work that can be reconstructed, reviewed and governed.
