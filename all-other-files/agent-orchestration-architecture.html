Saple Bridge Agent Orchestration

A repository-grounded architecture for turning Saple's current swarm, PTY, context, and MCP capabilities into a durable local orchestration engine.

Scope: saple-bridge and sibling saple-mcp working trees. No implementation changes are included.

Executive decision

Saple-MCP is a strong context/control-plane foundation, but it is not currently an orchestration engine.

Integrate Saple-MCP into the Saple-Bridge repository and product, but run orchestration as an automatically managed, per-workspace engine process.

MCP is an adapter into that engine. It is not the engine's internal state machine.

Bridge UI ──────────────────┐
External MCP stdio clients ─┼──> managed Saple engine
Headless automation ────────┘       ├─ orchestration state machine
                                    ├─ process supervisor
                                    ├─ durable workspace state
                                    └─ scoped agent runtime tools
  • One authoritative mutation owner per workspace.
  • Headless execution and UI-independent recovery.
  • A common engine for Bridge, MCP clients, and automation.
  • Ordered events and reconnectable observation.

Investigation method

The analysis traced the actual execution, persistence, protocol, and process-management paths in both repositories. Two independent architecture investigations were run:

  1. An embedded Rust kernel optimized for the smallest interface and maximum direct reuse.
  2. A standalone engine optimized for durability, single-writer state, headless use, and multi-client access.

The runtime did not expose a subagent model selector, so the requested GPT-5.6 Terra High label could not be guaranteed. Official MCP documentation was checked because the implementation advertises an older protocol revision.

Saple-MCP

Rust binary/library serving one project per process through stdio JSON-RPC and a custom HTTP bridge. Its working tree contains substantial modified and untracked feature work that should be checkpointed before migration.

118 tests passed · 0 failed

What Saple-MCP does well

  • Local-first JSON/Markdown state under .saple/.
  • Path containment and atomic replacement.
  • Shared handlers for stdio and HTTP.
  • Memory, tasks, incidents, agents, runs, artifacts, mappings, permissions, prompts, and resources.
  • Structured errors and strong integration coverage.
  • A library entry point below the executable.

Why it is not an orchestrator yet

  • Swarm access is read-only.
  • Agent tools store registry metadata but never launch agents.
  • Run tools record timelines but never execute work.
  • Permissions are advisory.
  • No durable scheduler, process supervisor, retries, deadlines, recovery, revision checks, attempt identities, or structured runtime reporting.

Tool surface drift

The documentation says 58 tools. The implemented catalog contains 59 because add_task_event is implemented and tested but omitted from the documented count. 

12 memory + 6 task + 1 swarm + 10 incident + 6 run
+ 6 context + 6 agent + 5 artifact + 3 mapping + 4 permission
= 59 tools

The catalog should remain a compatibility adapter. It is too CRUD-oriented to be the internal orchestration interface.

Where orchestration lives today

The real scheduler is renderer-owned. swarmStore.ts launches agents, scans dependencies, propagates failure, enforces a parallel cap, and persists state. terminalStore.ts interprets PTY output and exits. pty.rs owns provider validation and OS processes.

Existing capabilities

  • Static DAG scheduling and cycle checks.
  • Parallelism limit and dependency-based launching.
  • Review gates, pause/resume/stop/relaunch, and transitive blocking.
  • Provider/model selection, prompts, skills, context files, mailboxes, and handoffs.
  • Attempt-like scoped completion markers and PTY exit fallback.
  • Basic restart reconciliation.

Structural limits

  • Bridge closure or restart loses orchestrated PTYs.
  • Recovery marks lost agents failed rather than resuming them.
  • Lifecycle state is duplicated across renderer state and several files.
  • Completion relies primarily on terminal text parsing.
  • Parallel editing agents share one checkout unless manually constrained.
  • Races are patched across several modules instead of prevented at one seam.

Cross-process correctness finding

Atomic rename prevents partial reads. It does not prevent lost updates between Bridge and one or more MCP processes.

  1. Bridge reads revision A.
  2. Saple-MCP reads revision A.
  3. Bridge writes A+B.
  4. Saple-MCP writes A+C.
  5. B is lost even though both writes were atomic.

Saple-MCP's lock registry is process-local. Bridge owns a separate registry. The file watcher reduces stale overwrites but cannot serialize this race.

The fix is a single mutation owner, not more watchers.

MCP standards finding

Saple-MCP advertises protocol 2024-11-05. The official current revision is 2025-11-25: versioning and lifecycle.

The standard HTTP transport is Streamable HTTP. The existing /tools/:toolName REST bridge is not MCP Streamable HTTP.

MCP Tasks can represent long-running and input-required work, but remains experimental and client support varies. It should be an optional adapter feature, not the durable workflow implementation.

  • Upgrade negotiation and schema support.
  • Add a real Streamable HTTP /mcp endpoint where needed.
  • Retain stdio compatibility.
  • Keep orchestration correct without MCP Tasks support.

Two independent designs

Embedded Bridge kernel

A deep Rust module inside Tauri owns commands, snapshots, events, scheduling, and PTY correlation.

Wins: smallest initial diff, direct PTY/keychain reuse, no IPC.

Loses: no headless operation, UI crashes interrupt work, external MCP needs Bridge or a competing owner.

Managed workspace engine

A per-workspace process owns scheduling, execution, state, and events. Bridge and MCP clients are command/event clients.

Wins: single writer, headless use, crash isolation, common external interface.

Costs: lifecycle, discovery, authentication, versioning, and detached-process QA.

For UI-bound swarms, embedding is adequate. For extremely capable and durable orchestration, the managed engine is stronger.

Recommended architecture: built-in managed engine

Use the standalone runtime design while eliminating the sibling-repository dependency. 

saple-bridge/
  crates/
    saple-engine/       # current saple-mcp + orchestration owner + adapters
  src-tauri/            # desktop shell and interactive terminals
  src/                  # React projection and operator controls
  • Keep the saple-mcp binary name and compatibility behavior.
  • Run exactly one engine per workspace.
  • Bridge starts, discovers, authenticates to, and updates it invisibly.
  • stdio mode connects to the owner rather than opening another writer.
  • Do not create a separate shared-core crate until two real in-repository consumers require it.

The deep orchestration interface

submit(spec, requestId) -> WorkflowRef

command(
  workflowId,
  expectedRevision,
  requestId,
  command
) -> WorkflowRef

observe(
  workflowId,
  afterSequence
) -> SnapshotAndEvents

command covers pause, resume, cancel, approve, reject, retry, steering, messages, human input, and validated graph extension. Provider arguments, queue internals, and storage shapes remain hidden.

Typed errors

InvalidPlan, InvalidTransition, Conflict, NotFound, Unauthorized, PermissionDenied, ApprovalRequired, Capacity, ProviderUnavailable, LaunchFailed, Persistence, StorageCorrupt, and Unavailable.

Canonical domain model and invariants

Workflow

Mission, immutable initial specification, revision, policies, steps, event sequence, and approvals.

Step

Dependencies, runtime configuration, approval/retry/write-isolation policy, and current outcome.

Attempt

Immutable launch identity with provider/model, process correlation, timestamps, artifacts, exit, and outcome.

Event & capability

Monotonic committed events plus short-lived attempt-scoped identities for child agents.

Core invariants

  • Canonical project roots; unique IDs; all dependencies exist; graphs remain acyclic.
  • Only the engine actor changes lifecycle state.
  • Retries create new attempts; stale attempt events are ignored and audited.
  • Starting plus running attempts never exceed the configured cap.
  • Paused workflows launch nothing; cancellation persists before teardown.
  • Commands are idempotent and revision-checked.
  • Permission checks occur immediately before engine-mediated side effects.

Crash-safe mutation order

  1. Authenticate and validate capability.
  2. Deduplicate request and check revision/invariants.
  3. Persist state, events, and pending side-effect intent atomically.
  4. Execute the side effect.
  5. Feed the result back through the same actor and persist the transition.
  6. Schedule newly ready steps and publish only committed events.

Storage strategy

Do not add SQLite in the first orchestration change. Start with one engine owner and one atomic document per workflow: 

.saple/orchestration/<workflow-id>.json

It contains revision, graph, attempts, events, idempotency keys, pending side effects, approvals, and messages. Large logs and artifact bodies stay separate.

Existing swarm, run, run-event, and session files become compatibility projections rather than competing writable sources. Add SQLite only when rewrite cost, event volume, or query needs become measurably limiting.

Runtime, transports, and structured agents

A private AgentRuntime seam is justified by a real production process supervisor and a deterministic fake used in state-machine tests. Avoid a provider plugin framework until a second genuinely different implementation exists.

Transport adapters

  • Bridge: typed wrapper over authenticated local Streamable HTTP.
  • MCP stdio: connector to the workspace owner.
  • MCP HTTP: standard Streamable HTTP adapter.
  • Headless CLI: workflow submit, command, and observe.

Scoped child-agent tools

Each runtime session is bound to one attempt. Keep scoped terminal markers and PTY exit fallback for providers that cannot call MCP.

Parallel editing

Read-only steps may share the checkout. Editing steps should use engine-owned Git worktrees, record their base commit, and feed results into an explicit integration/review step. Use Git; do not build a custom merge engine.

Security posture

Reuse

  • Path containment, provider allowlisting, safe model validation, keychain secrets, atomic writes, and process-tree termination.

Correct

  • Enable managed-engine authentication by default.
  • Keep operator credentials outside the project.
  • Issue short-lived attempt-scoped child capabilities.
  • Authorize every workflow mutation and event query.
  • Use provider-native sandboxing and worktree isolation for direct CLI execution.
  • Do not claim that advisory application rules sandbox a fully trusted provider CLI.
  • Fail closed on corrupt state, invalid capability, and version mismatch.

Implementation roadmap

Phase 0 · Checkpoint and baseline

Checkpoint Saple-MCP's dirty tree, separate existing feature work, correct the tool count, preserve the 118-test baseline, and add compatibility fixtures.

Phase 1 · Repository integration

Move Saple-MCP into Bridge without behavior changes. Update sidecar staging, Cargo paths, CI, and release packaging while preserving the binary name.

Phase 2 · Managed engine lifecycle

Add serve/connect modes, exclusive workspace ownership, authenticated discovery, dynamic endpoint selection, version handshake, and current MCP Streamable HTTP.

Phase 3 · Rust state machine

Port scheduler behavior and tests, add typed commands/snapshots/events/revisions/attempts, persist before side effects, and make Bridge a projection client.

Phase 4 · Reliability

Move orchestrated process ownership to the engine. Add retry, deadline, cancellation, recovery, durable approval/input states, output limits, and worktrees.

Phase 5 · Structured agent runtime

Add scoped runtime MCP sessions, structured reports, durable messages/artifacts/steering, and retain markers as fallback.

Phase 6 · Source-of-truth cutover

Make the engine authoritative, migrate or project legacy files, remove renderer scheduling and duplicate lifecycle/cycle/file logic, and route mutations through the owner.

Phase 7 · Advanced orchestration

Only after the durable static path: dynamic subtasks, nested teams, conditional branches, budgets, fair scheduling, and richer artifact validation.

Reuse and delete

Reuse

  • Bridge provider validation, process-tree handling, keychain, markers, scheduler tests, notifications, and Git review.
  • Saple-MCP containment, canonical storage helpers, structured errors, context, tasks/runs/events/artifacts, prompts/resources, stdio compatibility, Axum, and integration tests.

Delete after cutover

  • Renderer scheduler and launch logic.
  • Terminal-driven swarm state mutation.
  • Duplicate session lifecycle persistence and cycle validation.
  • Duplicate path/lock/memory implementations.
  • Independent mutation paths bypassing the engine.
  • Custom HTTP business routes if no consumer remains.

Principal risks and mitigations

Risk Mitigation
Dirty-tree migration Checkpoint first; make the repository move mechanical and separately reviewable.
Split-brain workspace owners OS-level exclusive ownership, discovery, version handshake, and connector mode.
Crash around process spawn Persist attempt and side-effect intent first; reconcile pending intents at startup.
Stale attempt completion Attempt-scoped callbacks and capabilities.
Orphan descendant processes Retain Job Object/process-group teardown and verify per OS.
MCP compatibility Retain stdio, negotiate versions, preserve binary/config identity, migrate additively.
Permission overclaim Separate engine authorization from OS/provider sandboxing.
Storage growth Measure; replace storage behind the same interface only when needed.

Acceptance criteria for the first capable release

  • Exactly one engine owner per workspace.
  • Bridge can close and reopen without losing workflow history.
  • Commands are idempotent and revision-checked.
  • Every process event is attempt-scoped.
  • Rust owns DAG scheduling, gates, pause/resume/cancel, retry, and capacity.
  • Crashes between intent and side effect are recoverable.
  • Bridge is a command/event client, not a scheduler.
  • stdio MCP clients reach the same owner.
  • Current negotiation and Streamable HTTP work where advertised.
  • Child mutation tools use scoped capabilities.
  • Editing agents can use isolated worktrees.
  • Existing context data remains readable throughout migration.

Deliberate non-goals

Do not initially add Kafka/NATS/RabbitMQ, Temporal, gRPC, Postgres, multi-machine scheduling, a provider marketplace, distributed leases, a custom Git merge engine, or SQLite before measurement demonstrates the need.

The shortest reliable path is one local engine, one writer, a three-operation interface, standard MCP adapters, and the existing provider/process capabilities reused behind it.
